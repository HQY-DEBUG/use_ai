# 需求解析与执行

用户输入：$ARGUMENTS

## 执行步骤

### 第零步：歧义检查

仔细阅读用户输入，判断是否存在不明确之处：

- 目标文件/模块不确定
- 实现方式有多种可能且差异显著
- 关键参数缺失（如版本号、文件名、字段名等）

若存在歧义，**立即提问，不要假设，不要继续执行**：

```
在解析前需确认以下问题：
1. <问题1>
2. <问题2>
```

收到用户回答后，将答案整合进需求，再进入第一步。

### 第一步：解析需求

仔细阅读用户输入（含第零步补充的信息），结合当前项目上下文，将其拆解为具体可执行的子任务。

输出格式如下（严格按此格式，不要省略）：

---
**需求解析结果**

原始输入：`<用户输入原文>`

拆解为以下任务：
1. **[文件/模块]** 具体操作描述
2. **[文件/模块]** 具体操作描述
...

执行顺序：按上述编号顺序执行
---

### 第二步：逐步执行

按解析出的任务列表，逐条执行：
- 每完成一个任务，标注 `✅ 任务N 完成`
- 涉及文件修改的，先读取再修改
- 所有实质性修改完成后执行 git commit

### 注意事项

- 解析时结合已打开的文件和项目结构理解上下文
- 不要添加用户未要求的额外功能
- 保持最小修改原则
